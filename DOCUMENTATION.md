#### `search.h` / `search.c`

**Назначение:** точный и нечёткий поиск по индексу.

**Структуры:**

- `SearchResult`: `doc_id`, `title`, `score`
- `SearchResults`: `results` (Vector топ-10), `total`, `time_ms`
- `FuzzyCandidate`: `term`, `distance`, `postings`

**API:**

| Функция | Описание |
|---------|----------|
| `intersectPostings(lists[], n)` | AND по `doc_id` (пересечение posting lists) |
| `search(idx, query)` | Токенизация запроса → lookup → intersect → топ-10 |
| `fuzzyFindCandidates(idx, term, max_distance)` | Обход дерева + Левенштейн ≤ max |
| `fuzzySearch(idx, query, max_distance)` | Fuzzy по каждому слову запроса, ранжирование |
| `printResultsText` / `printResultsJSON` | Вывод |
| `freeSearchResults` | Освобождение |

**Точный поиск:**

1. Запрос режется на токены (как в preprocess: lower, alnum/`_`, len > 2).
2. Для каждого токена — `lookupTerm`.
3. Если хотя бы один терм отсутствует — пустой результат.
4. `intersectPostings` — документы, есть во **всех** списках.
5. Топ-10: сортировка по `doc_id`, `score` = число токенов в запросе.

**Fuzzy:**

- Расстояние Левенштейна (DP, ограничение длины строк).
- `score = matched_terms * 10 - avg_distance` (как в README).
- Сортировка по `score` убыв., затем `doc_id`.

## 4. Вопросы проверяющего и ответы (к защите)

### Теория и выбор структур

**1. Что такое инвертированный индекс? Чем отличается от прямого?**

**Ответ:** Инвертированный индекс хранит отображение **слово → список документов**, где это слово встречается (posting list). Прямой индекс — **документ → слова** (как в исходном тексте). Для поиска по запросу из нескольких слов инвертированный индекс удобен: быстро находим списки по каждому терму и объединяем их (у нас — пересечение для AND). Прямой индекс пришлось бы сканировать все документы.

---

**2. Зачем три дерева? В чём разница AVL, RB и B-tree?**

**Ответ:** Все три — сбалансированные структуры для словаря **терм → posting list**, но с разной «жёсткостью» баланса и формой узла:

| | AVL | Red-Black | B-tree (t=3) |
|---|-----|-----------|----------------|
| Баланс | \|h(left)−h(right)\| ≤ 1 | Высота ≤ 2·log₂(n+1) | Узел: от t−1 до 2t−1 ключей |
| Операция при переполнении | Повороты (≤2) | Перекраска + повороты | Split узла |
| Типичное применение | Строгий баланс, быстрый поиск | `std::map`, ядро Linux | ФС, СУБД, большие данные |

Лабораторная сравнивает их на одной задаче (индексация + поиск), чтобы увидеть разницу в константах на реальных объёмах.

---

**3. Почему для словаря в памяти часто берут RB-tree, а не AVL?**

**Ответ:** RB-tree допускает чуть большую высоту, зато при вставке/удалении обычно **меньше поворотов** — дешевле при частых обновлениях индекса. AVL строже по балансу → чуть меньше сравнений при поиске, но дороже вставка. Для «живого» индекса с частыми добавлениями документов RB часто предпочтительнее; AVL — когда поиск доминирует и обновлений мало.

---

**4. Когда B-tree выигрывает AVL/RB?**

**Ответ:** Когда данные **не помещаются в RAM** или читаются **страницами** (диск, SSD): в одном узле B-tree много ключей → меньше обращений к памяти/диску, лучше локальность в кэше. В нашей лабе B-tree в RAM с маленьким t=3 может проигрывать RB из‑за массивов в узле и split’ов на малых n.

---

**5. Что такое posting list? Как из нескольких слов получается список документов?**

**Ответ:** Posting list — `Vector` из `PostingEntry` (`doc_id`, `title`) для одного терма. Запрос «python list sort» → три lookup в дереве → три списка → **пересечение** по `doc_id` (`intersectPostings`) → документы, где есть **все** три слова. Затем из пересечения берётся топ-10.

---

**6. Что означает AND-семантика? Что если слова нет в индексе?**

**Ответ:** AND = документ должен содержать **каждый** токен запроса. В `search.c`: если `lookupTerm` для любого токена вернул `NULL` или пустой список — сразу **пустой результат** (пересечение невозможно). Это строгая семантика, как в поиске «все слова обязательны».

---

### AVL

**7. Инвариант AVL? Как измеряется баланс?**

**Ответ:** Для каждого узла разница высот левого и правого поддерева **не больше 1 по модулю**: |balance| = |h(left) − h(right)| ≤ 1. В коде хранится `height` узла; balance factor = h(left) − h(right). При нарушении — повороты (single/double).

---

**8. Сколько поворотов максимум после одной вставки?**

**Ответ:** **Не больше двух** (один двойной поворот = два single). После вставки баланс нарушается только на пути от листа к корню; на каждом уровне достаточно одного исправления, цепочка не «протекает» выше второго поворота.

---

**9. Сложность поиска/вставки в AVL?**

**Ответ:** **O(log n)** по числу уникальных термов n, потому что высота AVL **O(log n)** (константа ~1.44·log₂ n). Поиск — спуск по высоте; вставка — спуск + O(1) поворотов на пути.

---

### Red-Black

**10. Пять свойств RB-дерева (CLRS).**

**Ответ:**

1. Каждый узел красный или чёрный.
2. Корень — чёрный.
3. Все листья (NIL/sentinel) — чёрные.
4. У красного узла оба ребёнка чёрные (нет двух красных подряд).
5. Для каждого узла все пути до листьев содержат одинаковое число **чёрных** узлов.

Отсюда высота ≤ 2·log₂(n+1).

---

**11. Зачем sentinel `nil`?**

**Ответ:** `nil` — общий **чёрный фиктивный лист**: у «настоящего» листа `left = right = parent = nil`. Не нужно проверять `NULL` при поворотах и в `rb_insert_fixup` — меньше ветвлений, код как в учебнике CLRS. Все «пустые» ссылки указывают на один объект.

---

**12. Почему корень после вставки перекрашивается в чёрный?**

**Ответ:** Во время fixup новый узел может стать **красным корнем** (свойство 2 нарушено: корень должен быть чёрным). В конце `rb_insert_fixup` явно: `T->root->color = RB_BLACK` — восстанавливаем инвариант корня без лишних поворотов.

---

**13. Сравните высоту RB и AVL при том же n.**

**Ответ:** AVL **ниже** (строже баланс): h_AVL ≈ 1.44·log₂ n. RB допускает до ~2·log₂(n+1) — примерно в 1.4 раза выше в худшем случае. Зато RB реже крутит дерево при вставке. На практике разница по времени поиска на средних n часто в пределах константы.

---

### B-tree

**14. Минимальная степень t = 3. Сколько ключей максимум в узле?**

**Ответ:** t = 3 → в узле от **t−1 = 2** до **2t−1 = 5** ключей (`BTREE_MAX_KEYS = 5`). Детей у внутреннего узла до **2t = 6**. При 6-м ключе в узле (переполнение) — split.

---

**15. Чем split отличается от поворота в AVL?**

**Ответ:** **Поворот** локально переставляет 2–3 узла BST, не меняя число ключей в дереве. **Split** делит переполненный узел на два и **поднимает средний ключ** в родителя; может каскадом раздувать корень (новый корень с одним ключом и двумя детьми). Это операция для **ветвистого** дерева, а не для бинарного.

---

**16. Почему ключи в массивах узла, а не по одному в BST-узле?**

**Ответ:** B-tree создан для **пакетного** хранения ключей в одной «странице» (блок памяти/диска): меньше указателей, лучше sequential access, меньше высота дерева. В BST один ключ на узел — много указателей и уровней; для диска это дорого.

---

**17. Поиск в B-tree: как выбирается ребёнок?**

**Ответ:** В узле ищется индекс `i` — первый ключ, который ≥ искомому (`find_key_index`). Если ключ совпал — найден. Если лист — нет. Иначе спуск в **`children[i]`** (все ключи в поддеревье i меньше keys[i] или между keys[i−1] и keys[i] по правилам B-tree). В нашей реализации: цикл по узлам, сравнение `strcmp`.

---

### Реализация проекта

**18. Почему posting list — `Vector`, а не связный список?**

**Ответ:** По ТЗ переиспользуется **lab3/vector**; плюс массив даёт **локальность** при пересечении (последовательный обход), проще клонирование (`clonePostingList`). Для лабораторного масштаба append O(1) амортизированно достаточно.

---

**19. Что делает `appendPosting`, если тот же `doc_id` уже в конце списка?**

**Ответ:** **Ничего не добавляет** (ранний return). Защита от дублей, когда один и тот же документ индексируется повторно тем же термом (например, несколько вхождений слова в title+body дают один токен в JSONL, или повторный проход). Иначе posting list раздувался бы одинаковыми `doc_id`.

---

**20. Формат `index_<type>.txt`. Можно ли перенести между типами деревьев?**

**Ответ:** Текстовый формат: заголовок `LAB5IDX1`, строка типа (`avl`/`rb`/`btree`), блоки «длина ключа → ключ → число posting → строки doc_id+title», маркер `END`. **Между типами не переносится** без пересборки: `loadIndex` проверяет, что тип в файле совпадает с `--type`. Логически данные те же, но загрузка идёт через insert в выбранное дерево — можно было бы конвертировать, отдельного конвертера нет.

---

**21. Как парсится JSONL в C без cJSON? Ограничения?**

**Ответ:** Поиск подстрок `"doc_id"`, `"title"`, `"tokens"` и ручной разбор: `atoi` для id, копирование title между кавычками, для tokens — обход массива в `[...]` по кавычкам. **Ограничения:** нет полноценного JSON (экранирование, вложенность, unicode); title с кавычками внутри может сломаться; нестандартные поля игнорируются. Для контролируемого `docs.jsonl` после `preprocess.py` достаточно.

---

**22. Как работает `intersectPostings`? Сложность?**

**Ответ:** Берётся первый список; для каждого `doc_id` проверяется наличие во **всех** остальных (линейный проход по каждому списку). При совпадении — `appendPosting` в результат. **Сложность:** O(n₁ · (n₂ + … + nₖ)) в худшем случае; при отсортированных по doc_id списках можно O(Σ nᵢ) слиянием — в нашей реализации сортировка posting lists не гарантирована, используется простой вариант.

---

**23. Топ-10 и `score` в точном поиске?**

**Ответ:** После intersect все документы копируются в временный `Vector` `SearchResult`, **сортируются по `doc_id`** (`qsort`), в ответ попадают **первые 10**. `total` = полный размер пересечения. `score` = **число токенов в запросе** (все совпали → одинаковый score; ранжирование по релевантности не TF-IDF, а по doc_id). `time_ms` — разница monotonic time до/после поиска.

---

**24. Алгоритм fuzzy search: кандидаты и score?**

**Ответ:**

1. Запрос токенизируется как в точном поиске.
2. Для каждого слова: `fuzzyFindCandidates` — **обход всего дерева** (`traverseIndex`), для каждого ключа в индексе считается **Левенштейн**; если ≤ `max_distance` — кандидат с его posting list.
3. По документам агрегируется: сколько слов запроса «покрыто» кандидатами, минимальная дистанция на слово.
4. **score = matched_terms × 10 − avg_distance** (из README).
5. Сортировка по score убыв., топ-10.

---

**25. Сложность fuzzy при полном обходе дерева?**

**Ответ:** Пусть **m** — число уникальных термов в индексе, **L** — средняя длина ключа, **d** — max_distance. На одно слово запроса: обход **O(m)** узлов/ключей, на каждый ключ Левенштейн **O(L²)** (DP в реализации). Итого **O(m · L²)** на слово; на запрос из w слов — **O(w · m · L²)**. Это дорого; для продакшена нужны другие структуры (BK-tree, trigram index).

**27. Почему кривые 50k/200k/500k могут быть не монотонны?**

**Ответ:** Влияют **кэш CPU** (повторные прогоны быстрее), **фрагментация heap**, разный объём уникальных термов, **фоновая нагрузка ОС**, размер posting lists (длинные списки у частых слов замедляют intersect). На малых данных доминируют константы, а не асимптотика.

---

**28. Почему B-tree может быть быстрее/медленнее RB на малых n в RAM?**

**Ответ:** При **малом n** высота мала у всех; решают **константы**: у B-tree — циклы по ключам в узле, split, массивы; у RB — один ключ, меньше памяти на узел, проще сравнение. t=3 маленький — выгода B-tree по кэш-страницам почти не проявляется. Поэтому на тестовых 3 документах разница нулевая; на 500k — смотрят замеры.

---

**29. Что такое `ru_maxrss`? Почему на Windows `-1`?**

**Ответ:** **Maximum resident set size** — пик физической памяти процесса (на Linux в KB). В `main.c` `peak_rss_kb()` вызывает `getrusage` только под **`__linux__`**; на Windows API не подключён → возвращается **-1** («недоступно»). Для отчёта на Windows нужен Task Manager, GetProcessMemoryInfo или прогон bench в WSL/Linux.

### Углублённые / «подковырки»

**35. Можно ли OR вместо AND? Как изменить `search.c`?**

**Ответ:** Да. Вместо `intersectPostings` — **объединение** по `doc_id` (union): обойти все списки, собрать уникальные doc_id (hash set или сортировка+merge). Score можно считать числом совпавших термов. AND в коде: если любой lookup пуст — выход; для OR пустой список просто не добавляет документов, остальные слова всё ещё дают результаты.

---

**36. Почему fuzzy — полный обход, а не BK-tree?**

**Ответ:** Задание lab: fuzzy через **обход дерева + Левенштейн**. BK-tree / индекс по k-граммам — промышленные оптимизации; их не требовали. Полный обход **простой и корректный** для учебного объёма, но не масштабируется на миллионы термов.

---

**37. Утечки памяти: что освобождается?**

**Ответ:**

- `freeAVLTree` / `freeRBTree` / `freeBTree`: рекурсивно узлы, `free(key)`, `vectorFree(postings)`.
- `freeIndex`: соответствующее `free*Tree` + `free(Index)`.
- `freeSearchResults`: `vectorFree(results)` + `free(SearchResults)`.
- `loadIndex` после ошибки — частично может утекать при обрыве посередине (в учебном коде без rollback); при успешном пути всё в дереве.
- `fuzzyFindCandidates`: Vector кандидатов освобождает вызывающий (`vectorFree(cl)`); postings **не копируются** — указатели в дерево, не free в кандидатах.

---

**38. Потокобезопасность параллельного поиска?**

**Ответ:** **Нет.** Один `Index` в памяти без блокировок; `search`/`fuzzySearch` только читают дерево, но одновременная **запись** (insert) и чтение не поддерживаются. Параллельные чтения теоретически возможны, если дерево не меняется, но в C без `const`-корректности и барьеров это не гарантировано — для лабы один поток.

---

**39. Очень частый терм (стоп-слово)?**

**Ответ:** У него **огромный posting list** → `lookup` быстрый O(log n), но **intersect** перебирает все doc_id из самого длинного списка → поиск тормозит. В preprocess слова длиной ≤2 отбрасываются, но «the», «and» на реальном SO всё равно частые. Решение в продакшене: стоп-лист, не индексировать частые слова, или skip intersect с самым частым термом последним.

---

**40. Наш B-tree vs B+-tree в СУБД?**

**Ответ:** В **B+-tree** ключи **только в листьях**; внутренние узлы — разделители; листья связаны **linked list** для range scan. В нашей реализации **классический B-tree**: ключи и в внутренних узлах, posting list привязан к ключу в узле, нет цепочки листьев. Для полнотекстового словаря в RAM учебный вариант достаточен; СУБД использует B+ для последовательного чтения диапазонов с диска.

## 5. Краткая шпаргалка для ответа «как устроен проект»

> CSV токенизируется в Python → JSONL. C читает JSONL и для каждого токена вставляет `(терм → posting list)` в выбранное дерево. Posting list хранит пары `(doc_id, title)`. Поиск: токены запроса → lookup в дереве → пересечение списков по doc_id → топ-10. Fuzzy: обход всех ключей дерева, Левенштейн ≤ d, объединение и ранжирование. Индекс можно сохранить в текстовый файл и загрузить без повторной индексации.
